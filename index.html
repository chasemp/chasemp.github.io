debian rpaf fools mod_access

Tuesday, August 06, 2013 » debian, rpaf

Preamble

If you have a reverse proxy in front of apache you need a way to
persist the original host IP for logging, etc. rpaf does
this. Essentially handles letting apache see the proxy provided
field of x-forward-for so things are handled in a sane manner.

Unfortunately this seems to have problems working with mod_access.

Debian report
Debian report
ubuntu report
poundlist report

Everyone needs to authenticate

Deny from all, but if it is a valid user then it will trigger Satisfy Any.


    Order deny,allow
    Deny from all
    AuthType Basic
    AuthName "Authentication Required"
    AuthUserFile "/etc/htpasswd/.htpasswd"
    Satisfy Any
    Require valid-user

Everyone needs to authenticate except for internal hosts

Deny from all but Satisfy Any is triggered by a '192.' host or valid auth.


    Order deny,allow
    Deny from all
    Allow from 192
    AuthType Basic
    AuthName "Authentication Required"
    AuthUserFile "/etc/htpasswd/.htpasswd"
    Satisfy Any
    Require valid-user

--> X-Forwarded-For: 99.9.236.147

In Apache my host shows up as: 99.9.236.147.

Hmmmm....Access Granted

Apache's ACL system is still treating my requests as if they were sourced from 192.168.0.101.

The configuration from 'Everyone needs to authenticate except for internal' should allow
only internal hosts to bypass auth. In this case it is allowing everyone because
Apache is not using the post-rpaf x-forward-for IP in its ACL lookup. It is using the
IP of the reverse proxy. This means everyone is allowed.

Workaround: Getting more specific with our allowances

Only allow specific internal ranges that do not include the reverse proxy referrer. 


    Order deny,allow
    Deny from all
    Allow from 192.168.1.
    AuthType Basic
    AuthName "Authentication Required"
    AuthUserFile "/etc/htpasswd/.htpasswd"
    Require valid-user
    Satisfy Any

This is not a great fix as it allows for a future re-IP to cause chaos.

Thoughts

The CSF pack if pretty powerfull and I have a general idea of the usage experience. I have
not tested this with Webmin or cPanel or any other wrapper. I wouldn't use them if I had
the choice anyways. I think this exists in a weird space as anyone who does operations

administration would do well to side step it and learn the underlying tools directly. I
appreciate what they have going here and I can tell a lot of thought went into it.
However, if I were talking to myself as I was starting out with a personal Debian server
I would still not recommend going this route.

Instead I might recommend:

  1. disable root account remote access
  2. lock SSH down to key based auth
  3. whitelist your IP
  4. basic ip tables setup
  5. avoid running sendmail, instead try nullmailer
  6. avoid running BIND
  7. use fail2ban
  8. Don't worry about portscanning, it's inevitable
  9. Try something like CloudFlare's free service
  10. Use Puppet and Git for deploying even small setups
  11. Backup up stuff outside of your server
  12. Use static site generators if feasible: jekyll or mynt

This is the basic approach that I prefer, but kudos to the CSF guys for putting
something out there to help people. I can see using this in a lab to quickly reconfigure
a dummy host for attacking or tutorials.

PEP8 Guide Updates

Thursday, August 01, 2013 » PEP8, Python

Preamble

PEP8 is the 'official' style guide for Python. The idea being if the Python
community can rally around a set of standards it makes the overall quality of
code in the community better. At least that's my idea.

The guide has now been updated. Here are some things I found notable.

Style Guide Exclusions

Then:

-Two good reasons to break a particular rule:

-1. When applying the rule would make the code less readable, even for
-   someone who is used to reading code that follows the rules.

Now:

+In particular: do not break backwards compatibility just to comply with
+this PEP!

+Some other good reasons to ignore a particular guideline:
+
+1. When applying the guideline would make the code less readable, even
+   for someone who is used to reading code that follows this PEP.

+3. Because the code in question predates the introduction of the
+   guideline and there is no other reason to be modifying that code.
+
+4. When the code needs to remain compatible with older versions of
+   Python that don't support the feature recommended by the style guide.

Line Characters

Then:

-Limit all lines to a maximum of 79 characters.

Now:

+Aim to limit all lines to a maximum of 79 characters, but up to 99
+characters is acceptable when it improves readability.

+For flowing long blocks of text with fewer structural restrictions
+(docstrings or comments), limiting the line length to 72 characters
+is recommended.

UTF-8 Is The Future

Then:


   -Code in the core Python distribution should always use the ASCII or
   -Latin-1 encoding (a.k.a. ISO-8859-1).  For Python 3.0 and beyond,
   -UTF-8 is preferred over Latin-1, see PEP 3120.

Now:


   +Code in the core Python distribution should always use UTF-8 (or ASCII
   +in Python 2).

   +  Note that in Python 3, ``unicode`` and ``basestring`` no longer exist
   +  (there is only ``str``) and a bytes object is no longer a kind of
   +  string (it is a sequence of integers instead)

Relative vs. Absolute Imports

Then:


   - Relative imports for intra-package imports are highly discouraged.
   -  Always use the absolute package path for all imports.  Even now that
   -  PEP 328 is fully implemented in Python 2.5, its style of explicit
   -  relative imports is actively discouraged; absolute imports are more
   -  portable and usually more readable.

Now:


   + Absolute imports are recommended, as they are usually more readable
   +  and tend to be better behaved (or at least give better error
   +  messages) if the import system is incorrectly configured (such as
   +  when a directory inside a package ends up on ``sys.path``)::
   +
   +    import mypkg.sibling
   +    from mypkg import sibling
   +    from mypkg.sibling import example
   +
   +  However, explicit relative imports are an acceptable alternative to
   +  absolute imports, especially when dealing with complex package layouts
   +  where using absolute imports would be unecessarily verbose::
   +
   +    from . import sibling
   +    from .sibling import example

Other Lessons Learned

  • Use 'def' and not named lambda's
  • Backwards compability is only expected for public interfaces
  • Use ''.Join() over concatenation with performance and portablity in mind
  • Use 'with' statements with local resources. try/finally is still ok.
  • Don't break old code just to apply the style guide on principle.
  • Python3.x does not allow mixing tabs and spaces

syn cookies

Preamble

When you're being hit with DDOS of any flavor it sucks mightily. Some of the old school attacks like
syn floods have been around so long there are decent defenses. One of the (seemingly) lesser known
defenses is syn cookies.

Who doesn't like cookies?

SYN cookies are particular choices of initial TCP sequence numbers by TCP servers.

Linux

root@<server>:~# cat /proc/sys/net/ipv4/tcp_syncookies
0

root@<server>:~# cat /proc/sys/net/ipv4/tcp_syncookies
1

Now syn cookies are enabled. This not a foolproof defense but can help. Remember this method of
enabling does not persist through a reboot.

Cisco IOS Zone Based FW

Global:

parameter-map type inspect global
    tcp syn-flood limit 20000

Zone:

parameter-map type inspect-zone ddos-detection
   tcp syn-flood rate per-destination 20000

NOTES:

SYN cookies are not without problems. They can cause latency, sometimes very noticeable latency.

postfix ssl weak ciphers

Vuln

Weak ciphers are detected as a level 4 vuln for most scanners.

If you have to deal with PCI you know all level 4 and 5 vulns must be resolved.

Resolution

/etc/postfix/main.cf

# TLS parameters
smtpd_use_tls=yes
smtpd_tls_mandatory_ciphers = high
smtpd_tls_mandatory_exclude_ciphers = aNULL, MD5, DES
smtpd_tls_security_level = encrypt
#Ensure TLSv1
# Preferred form with Postfix ≥ 2.5:
smtpd_tls_mandatory_protocols = !SSLv2, !SSLv3

Notes

Pre and post scans confirm all is well but...

Postfix is strange when it comes to SSL, and 

smtpd_tls_security_level = encrypt

implies

ssl_only_use_tls = true.

So if you have need for non-tls capable clients use caution.

using graphite events

Friday, May 17, 2013 » graphite

Preamble

Graphite events allow you to overlay vertical bars on graphs for correlation of events
and trending data. I know the etsy
guys use normal whisper files; presumabley with a 0.0xFacter and sum rollup instead of average. That
is a nice and reliable way to do it, but if you are looking for more flexbility using the
native ('experimental') events feature may be useful.

Posting an event:

#!/bin/bash
curl -X POST http://graphite/events/ -d '{"what": "did_stuff", "tags" : "fun,times"}'

Retrieving events:

curl -X GET http://graphite/events/

curl -X GET http://graphite/events/?tags=fun

Show me only stuff starting at an epoch time:

curl -X GET http://graphite/events/?from=1368778407

Getting lines on graphs

Now you have an event that shows up on http://graphite/events and is retrievable by curl.

Place it over your graph:

https://graphite/render?target=my.metic.path&target=drawAsInfinite(events('fun'))

You can also go two tags deep which is similar in logic to a SQL join. So the previous example
retrieves all 'fun' tagged events. Now we want to get 'fun' and correlate with 'times'.

https://graphite/render?target=my.metic.path&target=drawAsInfinite(events('fun', 'times'))

This returns all 'fun' tagged events that also have 'times' tags.

Where this is useful in real world:

#all syncs
https://graphite/render?target=my.metic.path&target=drawAsInfinite(events('code_sync'))

#Now code syncs from a specific user:
https://graphite/render?target=my.metic.path&target=drawAsInfinite(events('code_sync', 'user'))

simple iptables uses

Allow ssh from only 192 network (presumably your internal)

iptables -A INPUT -i eth0 -p tcp -s 192.168.0.0/16 --dport 22 -m state --state NEW,ESTABLISHED -j ACCEPT
iptables -A OUTPUT -o eth0 -p tcp --sport 22 -m state --state ESTABLISHED -j ACCEPT

Allow SCCP (VOIP Protocol)

iptables -A INPUT -i eth0 -p tcp --dport 1720 -m state --state NEW,ESTABLISHED -j ACCEPT
iptables -A OUTPUT -o eth0 -p tcp --sport 1720 -m state --state ESTABLISHED -j ACCEPT

Allow h.323 (VOIP)

iptables -A INPUT -i eth0 -p tcp --dport 2000 -m state --state NEW,ESTABLISHED -j ACCEPT
iptables -A OUTPUT -o eth0 -p tcp --sport 2000 -m state --state ESTABLISHED -j ACCEPT

Allow ICMP

iptables -A OUTPUT -p icmp --icmp-type echo-request -j ACCEPT
iptables -A INPUT -p icmp --icmp-type echo-reply -j ACCEPT

Allow DNS queries, etc

iptables -A OUTPUT -p udp -o eth0 --dport 53 -j ACCEPT
iptables -A INPUT -p udp -i eth0 --sport 53 -j ACCEPT

Allow mail sending

iptables -A INPUT -i eth0 -p tcp --dport 25 -m state --state NEW,ESTABLISHED -j ACCEPT
iptables -A OUTPUT -o eth0 -p tcp --sport 25 -m state --state ESTABLISHED -j ACCEPT
